# Scoring Rubrics (1-10 Scale)

All evaluators use these rubrics for consistency. Scores should be evidence-based, not aspirational.

## Commercial Score
| Score | Criteria |
|-------|----------|
| 1-2 | No clear revenue model. No identifiable customer. Charity/hobby only. |
| 3-4 | Possible revenue but unclear path. Small niche. Difficult monetization. |
| 5-6 | Viable revenue model. Identifiable customers. Competitive market but defensible angle. |
| 7-8 | Strong revenue potential. Clear differentiation. Multiple revenue streams possible. Growing market. |
| 9-10 | Exceptional market opportunity. Strong moat. Large TAM with clear path to SOM. Scalable revenue model. |

## Educational Score
| Score | Criteria |
|-------|----------|
| 1-2 | No educational value. No skill transfer. No learning opportunity for builder or users. |
| 3-4 | Minor learning for builder. No external educational value. |
| 5-6 | Moderate learning for builder. Some educational value for users or community. |
| 7-8 | Significant skill development. Clear educational impact on a community. Reusable knowledge. |
| 9-10 | Transformative learning opportunity. Democratizes knowledge. Creates new educational pathways. |

## Social Score
| Score | Criteria |
|-------|----------|
| 1-2 | No social benefit. Potentially harmful. Extractive. |
| 3-4 | Neutral social impact. Benefits only the builder. |
| 5-6 | Some community benefit. Positive but limited reach. |
| 7-8 | Clear social good. Addresses a real community need. Inclusive design. |
| 9-10 | High social impact. Addresses systemic issues. Empowers underserved communities. Aligns with SDGs. |

## Technical Feasibility Score
| Score | Criteria |
|-------|----------|
| 1-2 | Requires technology that doesn't exist. Fundamental unsolved problems. |
| 3-4 | Technically possible but extremely complex. Requires rare expertise. High uncertainty. |
| 5-6 | Feasible with effort. Known solutions exist but require significant integration. Moderate complexity. |
| 7-8 | Clearly feasible. Well-understood technology. Builder has most required skills. |
| 9-10 | Straightforward to build. Proven patterns. Builder has all required skills. Low technical risk. |

## Risk Level Score (inverted - higher = less risky)
| Score | Criteria |
|-------|----------|
| 1-2 | Critical risks with no mitigation path. Legal/regulatory blockers. Single point of failure with no alternative. |
| 3-4 | Significant risks. Multiple dependencies on external factors. Unclear regulatory landscape. |
| 5-6 | Moderate risks with known mitigations. Some external dependencies. Manageable uncertainty. |
| 7-8 | Low risk. Well-understood domain. Few dependencies. Clear path forward. |
| 9-10 | Minimal risk. Fully within builder's control. No regulatory concerns. Reversible decisions. |

## Resource Efficiency Score
| Score | Criteria |
|-------|----------|
| 1-2 | Requires massive investment with no staged approach. All-or-nothing. Years to any result. |
| 3-4 | High resource requirements. Needs significant funding or team. Long timeline to value. |
| 5-6 | Moderate resources needed. Can be staged. 3-6 months to initial value. |
| 7-8 | Lean execution possible. Solo or small team viable. Weeks to initial value. Low cost. |
| 9-10 | Minimal resources needed. Can start today. Nearly free. Quick to first result. |

## Opportunity Score (Overall)
Weighted average of all sub-scores, adjusted by evaluator consensus:
| Score | Interpretation |
|-------|---------------|
| 1-2 | **Pass** - Not worth pursuing. Major fundamental issues. |
| 3-4 | **Weak** - Significant concerns outweigh potential. Consider pivoting the concept. |
| 5-6 | **Moderate** - Has merit but needs refinement. Proceed with caution and validation. |
| 7-8 | **Strong** - Good opportunity. Clear path forward with manageable risks. Recommended. |
| 9-10 | **Exceptional** - Rare alignment of value, feasibility, and timing. Prioritize this. |

## Project Health Score (Existing Projects Only)
| Score | Criteria |
|-------|----------|
| 1-2 | Abandoned or broken. No tests. Severe tech debt. Outdated dependencies with vulnerabilities. |
| 3-4 | Functioning but fragile. Minimal tests. Significant tech debt. Documentation gaps. |
| 5-6 | Working state. Some tests. Moderate tech debt. Basic documentation exists. |
| 7-8 | Healthy codebase. Good test coverage. Clean architecture. Up-to-date dependencies. |
| 9-10 | Exemplary project. Comprehensive tests. Clean code. Full documentation. CI/CD. Active maintenance. |
